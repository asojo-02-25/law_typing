# law_typing

司法試験受験生向けのタイピング練習サイト。主要7科目の法令文を題材に、速度(keys/秒)・正タイプ率・苦手キーを可視化する実戦的な入力精度を鍛えることを目的としている。

## 開発背景

司法試験及び司法予備試験では、2026年度試験よりCBT(Computer Based Testing)式試験が導入予定である。これに伴い、タイピングの早さ、正確性、ブラインドタッチの可否は単なる入力作業の差に留まらず、思考を言語化し、文章を構成するために割ける時間の差に直結すると考えた。競合サイトを検索したところ、現行のタイピング練習サイトには法律実務や試験対策に特化した専門性の高いコンテンツが不足していることを感じた。そこで、法律関連の知識をインプットしながら実践的な打鍵能力を向上させることをコンセプトに、本サイトの開発を行った。また、初めての個人開発にあたり、モダンなフレームワークによる抽象化に頼るのではなく、ブラウザ標準技術（vanilla js）を直接扱うことでWeb開発の基礎原則を深く理解すること、およびPythonを用いた実務に耐えうるデータ変換パイプラインの構築を技術的な挑戦として設定した。

## 主な機能

- 主要7科目に対応(憲法、民法、商法、民事訴訟法、行政法、刑法、刑事訴訟法)
- 問題数と分野、プレイ時のキーボード表示有無を選択可能
- プレイはspace押下、プレイ停止はesc押下により可能(マウス不要)
- 問題には出題分野と出典(例 : 〇〇法〇〇条〇〇項)を明記
- 次に打つべきキーをハイライト表示
- 「し」→ "shi", "si", "ci" などの表記ゆれに対応
- プレイ履歴(keys/秒, 正タイプ率)を期間別にグラフ化し変化を可視化
- 外れ値(低速・低正答)を履歴集計から自動除外
- keys/秒、正タイプ率、ミスタイプキーを表示し、同時に過去データの平均と変化率を比較
- タイプした法律を履歴として表示し後に参照可能
- サイドバーにプレイ日数関連データや、過去全データから計算したデータを表示
- プレイデータはLocalStorage保存(サーバー送信なし)
- ダークモードとの切り替え可能
- データ生成時に旧仮名の促音表記(つ→っ)を辞書方式で正規化し、未対応候補を自動レポート

### 1. 複数の入力候補に対応する入力エンジンの自作
  
【背景・課題】複数のタイプ方法がある文字に対して、単一のタイプ方法しか受理しないような仕組みはユーザー体験を大きく低下させると考えた。例として、「かんない」とタイプする場合を考えたとき、「ん」を"n", "nn"どちらで解釈するかによって、"kannai", "kannnai"の2つの読み方が考えられる。このとき、"kan"を「かん」で確定してしまうと"kan/nnai"とタイプする際に本来正しくタイプできているのにミス判定になってしまう。
  
【解決】"kan"の段階では入力候補を確定せず(保留)、"kannn"まで入力されて初めて"ka/nn/na/i"の入力候補で確定する。このように候補の確定に「保留」という状態を採用することで複数の入力候補に対応できるようになった。

【結果】動的な入力判定ロジックを自作することで表記ゆれに対応した。

  
### 2. ゲーム終了時のアニメーション不具合の解決

【背景・課題】ゲーム終了時にゲーム画面の要素の高さや透明度を変更することで結果画面へスムーズな画面遷移を行っている。しかし、複数の要素を同時に動かすことから、いくつかの困難に直面した。

1. アニメーションのキャンセルが想定通りなされず、2回目以降のプレイ時に画面表示に不具合が生じた。
2. 複数の要素を同時に動かすため、他の要素との干渉により想定通り動作しないことがあった、用いるアニメーションによって正しく動作しなかった。

【解決】
1. アニメーション対象の要素のstyle指定をjsで改めて指定することで、保守性と引き換えに解決。
2. 動かす要素の構造を、htmlにおいて「枠」と「中身」に分解。枠の高さを変化と中身の透明度の変化を別のアニメーションに分けて実装することで干渉を防ぐことができた。

【結果】画面遷移関連の不具合に関して、原因を特定し解決策を導くことができた。

### 3. 実装コストと情報の正確性、学習価値のバランスを考慮したタイピングコンテンツの決定

【背景・課題】タイピングの練習をしながらインプットする知識として最も学習価値の高いものは「規範(裁判の判断基準となるもの。判例文に示される。)」であると考える。しかし、規範を判例文から抽出することはLLMを頼っても非常に時間的コストが大きく、また抽出の際に正確性が損なわれる可能性がある。また、規範は書籍には抽出されているが、書籍に掲載されている文章を用いるのは法的問題がある。

【解決】学習価値と情報の正確性、実装コストの観点から思考、調査するうち、法律の条文を法令APIとpythonスクリプトにより自動取得できることを紹介している記事を発見した。

【結果】主要7科目の法律の条文をタイピングコンテンツとして採用することで、学習価値と情報の正確性を担保しながら実装コストを最小化しつつ、pythonを用いた開発という新たな経験を積むことができた。

### 5. ユーザー体験および学習の質を維持するためのデータクレンジング

【背景・課題】取得した該当科目の法律の条文から、タイピングに適した条文のみをタイピングコンテンツとして採用した。しかし、一括取得した条文をすべて問題として採用すると、タイピングしづらい問題や学習価値の低い問題が含まれていた。

【解決】具体的には、「（以下「〇〇」という。）」といったかっこを含む条分や、以下の例のような参照を行う法律を、タイピングのしやすさ及び学習価値の観点からを削除する条件をpythonスクリプトに追加した。

  例) 行政手続法 第 2 条
    
    第2条 この法律において、次の各号に掲げる用語の意義は、当該各号に定めるところによる。 ← これが本来はタイピングコンテンツであるが、学習価値が極めて低い
    
      1. 法令法律、法律に基づく命令（告示を含む。）、条例及び地方公共団体の執行機関の規則（規程を含む。以下「規則」という。）をいう。
      2. 処分行政庁の処分その他公権力の行使に当たる行為をいう。
    
    …
【結論】タイピングのしやすさと法律上の学習価値を両立したデータセットが完成。

### 6. 法律用語に特化した高精度な「かな変換」の実装

【背景】タイピングコンテンツとして法律の条文を用いるうえで、法律特有の難解な語彙をいかに正確なタイピングデータに変換するかであった。しかし、当初用いていた汎用ライブラリは熟語の変換に弱く、「価額（かかく）」が「価額（かひたい）」と誤読されたり、「遺言（いごん）」などの法律専門用語の読みを正しく判定できない問題があった。

【解決】形態素解析エンジンに SudachiPy を採用し、語彙数が最も豊富な SudachiDict Full を導入。さらに、エンジンでも判別が難しい特定の法律用語（例: 「額」「遺言」等）については、解析後にカスタムマッピングを適用する二段構えの補正ロジックを構築した。

【結果】実戦的な司法試験対策に耐えうる、正確な問題セットの自動生成を実現しました。

## 技術スタック

- Frontend: HTML / CSS / JavaScript (vanilla js)
  - コアロジックはvanilla jsで作成。グラフ描画のみchart.jsを使用。理由は下記の2点。
  - 初めてのweb開発であったため、フレームワークを使用せず、基本的なDOM操作やイベントリスナーを理解するため。
  - シングルページのwebアプリ開発において、外部フレームワークの導入は学習コストが重く、オーバーエンジニアリングであると判断したため。
- Visualization: Chart.js
  - コアロジックの開発及び理解に集中するためグラフ描画にのみ外部ライブラリを使用。
  - 外部ライブラリのうち、特定にフレームワークに依存せずvanilla jsとの親和性が高いchart.jsを選択。
- Data source: e-Gov法令API v1
  - pythonスクリプトとあわせて現時点での最新の法律を一括取得でき、問題実装コストを最小化するため。
- Backend scripts: Python
  - 法令XMLのパース、形態素解析、JSON変換の自動化に最適であるため。法令改正にも強い。
- NLP: SudachiPy + SudachiDict Full
  - 法律用語を文脈に即して解釈できる正確性の高い変換ライブラリであるから。


## システム構成

### アーキテクチャ概要

```text
e-Gov API
	-> Step1: XML取得 (scripts/fetch_law_xml_step1.py)
	-> Step2: 本文正規化 (scripts/normalize_law_text_step2.py)
	-> Step3: かな変換 (scripts/convert_kana_step3.py)
	-> Step4: 最終整形/検証 (scripts/finalize_questions_step4.py)
	-> data/questions.json
	-> Web app (js/script.js)
```
